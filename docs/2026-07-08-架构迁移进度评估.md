# 架构迁移进度评估

评估日期：2026-07-08

## 评估依据

本评估基于当前 `docs/` 内已有架构文档和实施文档：

- `docs/agent需遵循架构约定.md`
- `docs/通用前后端架构文档.md`
- `docs/specs/2026-07-08-雷达架构样板改造设计.md`
- `docs/specs/2026-07-08-图谱架构样板改造设计.md`
- `docs/plans/2026-07-08-雷达架构样板改造实施计划.md`
- `docs/plans/2026-07-08-图谱架构样板改造实施计划.md`

核心评估标准：

1. 前端 `app` 只保留路由入口、参数读取和页面组合，业务代码进入 `features/<module>`。
2. 前端组件不直接散落请求逻辑，业务 API 进入 feature API 或 server service。
3. 后端采用模块化单体，业务进入 `backend/modules/<module>`。
4. 后端 router 只处理 HTTP 参数和响应，业务编排进入 service，数据来源进入 repository 或明确的依赖层。
5. 新迁移模块需要最小测试保护。
6. Radar/Graph 作为样板模块应优先满足统一响应、统一 request 和薄路由要求。

## 总体结论

当前迁移已从“样板建立阶段”进入“主要模块收敛阶段”。

Radar 和 Graph 已完成核心样板迁移，后端轻量模块也已经基本从 `backend/api` 收敛到 `backend/modules`。前端侧 Radar 和 Graph 的业务边界较清晰，Learn/Progress 已完成服务层抽取，但 Home、CMS、Labs 仍有较多业务逻辑直接留在 `app` 页面中。

综合判断：整体迁移进度约 **65%**。

分项估算：

| 维度 | 进度 | 判断 |
|---|---:|---|
| 后端模块化 | 80% | 主要 API 已迁入 `backend/modules`，`api/*_routes.py` 多数已变成兼容导出 |
| 前端 feature 化 | 55% | Radar/Graph 完成度高，Learn/Progress 部分完成，Home/CMS/Labs 待迁 |
| 统一响应与 request | 55% | Radar/Graph 达标，Chat/Content/Progress/Share 仍保留旧直接响应 |
| 测试保护 | 70% | 已覆盖 Radar、Graph、Chat、Learn、Progress、Content/Share 服务行为 |
| 文档与计划闭环 | 70% | Radar/Graph 有设计和计划；后续模块缺少独立计划文档 |

## 后端迁移状态

| 模块 | 当前状态 | 评估 |
|---|---|---|
| Radar | `backend/modules/radar` 已包含 repository、service、schemas、router；`api/radar_routes.py` 为兼容导出 | 达标 |
| Graph | `backend/modules/graph` 已包含 dependencies、service、schemas、router；`api/graph_routes.py` 为兼容导出 | 达标 |
| Chat | 已迁入 `backend/modules/chat`，拆出 schemas、service、router；保留 `/graph/chat` 响应格式 | 基本达标 |
| Content | 已迁入 `backend/modules/content`，原 API 变薄 | 基本达标，但仍是 stub 数据 |
| Progress | 已迁入 `backend/modules/progress`，原 API 变薄 | 基本达标，但后端仍是占位入口 |
| Share | 已迁入 `backend/modules/share`，原 API 变薄 | 基本达标，仍使用内存 store |
| Scheduler / Knowledge / Extractors | 仍按技术目录组织 | 暂不属于本轮 API 模块迁移范围 |
| Models Repository | `GraphRepository` 仍在 `backend/models/repository.py` | 符合 Graph 设计中的“暂不移动共享 repository”策略 |

后端当前主要风险：

- `backend/api` 仍作为兼容层存在，主应用 `main.py` 也仍从 `api.*_routes` 注册 router。短期可接受，长期应改为直接注册 `modules/*/router.py`。
- Radar/Graph 使用统一响应；Chat、Content、Progress、Share 仍返回旧直接结构。这是兼容策略的一部分，但后续需要明确哪些接口要切到统一响应。
- `ShareService` 仍使用内存 store，符合当前迁移目标，但不适合作为长期生产实现。

- `app/page.tsx`、`app/cms/page.tsx` 仍直接调用 `getDb()`，不符合文档中“app 只保留路由入口”的目标。
- Graph 仍有 `frontend/src/components/graph/*` 作为全局图谱组件。Graph 设计文档明确允许短期保留，因此不是当前阻塞项；后续严格收敛时再迁入 `features/graph/components`。

## API 与响应契约状态

| 接口族 | 响应状态 | 说明 |
|---|---|---|
| `/radar/*` | 统一响应 | 符合 Radar 样板 |
| `/graph/nodes`、`/graph/edges`、`/graph/subgraph` 等 | 统一响应 | 符合 Graph 样板 |
| `/graph/chat` | 旧直接响应 | 为保持 RAG Chat 兼容，暂不切统一响应 |
| `/content/*` | 旧直接响应 | 当前主要是 stub |
| `/progress/*` | 旧直接响应 | 后端占位；真实进度主要由 Next Route Handler 管理 |
| `/share/*` | 旧直接响应 / HTTPException | 可后续统一错误结构 |
| Next.js `/api/tracks`、`/api/modules`、`/api/lessons`、`/api/progress` | 旧直接响应 | 迁移中保留原前端契约 |

结论：统一响应已在样板模块落地，但还不是全项目默认契约。后续应先明确“外部公开 API”和“Next 内部 Route Handler”的响应策略是否必须一致。

## 剩余迁移任务优先级

### P0：收敛仍直接访问数据库的前端页面

目标文件：

- `frontend/src/app/page.tsx`
- `frontend/src/app/cms/page.tsx`
- `frontend/src/app/cms/editor/[...slug]/page.tsx`
- `frontend/src/app/cms/generate/page.tsx`
- `frontend/src/app/cms/publish/page.tsx`

建议做法：

1. 新增 `frontend/src/features/home/server/homeService.ts`，承载首页聚合查询。
2. 新增 `frontend/src/features/cms/server/cmsService.ts`，承载 CMS 列表和统计查询。
3. 页面只调用 service 并负责渲染。
4. 先写 characterization tests，再迁移实现。

### P1：统一轻量后端模块响应策略

目标模块：

- `backend/modules/chat`
- `backend/modules/content`
- `backend/modules/progress`
- `backend/modules/share`

建议先决策：

- Chat 是否继续保留旧响应，避免影响前端 RAG 调用。
- Content/Progress/Share 是否切到 `common.response.success_response` / `error_response`。
- `backend/api/*_routes.py` 是否继续作为长期兼容层，还是逐步改 `main.py` 直接注册 `modules/*/router.py`。

### P1：拆分 Learn service 的职责

当前 `features/learn/server/learningService.ts` 同时处理：

- tracks/modules/lessons API；
- Learn 页面聚合数据；
- design patterns API。

建议后续拆成：

```txt
features/learn/server/learningService.ts
features/learn/server/lessonPageService.ts
features/content/server/designPatternService.ts
```

拆分前先保持当前测试不变，避免行为变化。

### P2：收敛 Labs

当前 Labs 页面是静态数组。它不一定需要后端模块，但应至少进入：

```txt
frontend/src/features/labs/
├─ data/labs.ts
├─ components/
└─ types/
```

这样 `app/labs/page.tsx` 和 `app/labs/[id]/page.tsx` 只负责路由与渲染。

### P2：Graph 全局组件归位

当前以下组件仍在全局目录：

- `frontend/src/components/graph/GraphCanvas.tsx`
- `frontend/src/components/graph/NodeDetail.tsx`
- `frontend/src/components/graph/SearchBar.tsx`
- `frontend/src/components/graph/FilterPanel.tsx`

Graph 设计文档允许短期保留。后续若继续严格执行模块内聚，可迁入 `frontend/src/features/graph/components`。

## 建议的下一步执行顺序

1. 迁移 Home：新增 `features/home/server/homeService.ts`，移出 `app/page.tsx` 的 SQLite 聚合查询。
2. 迁移 CMS：新增 `features/cms/server/cmsService.ts`，移出 `app/cms/*` 中的 SQLite 查询和统计逻辑。
3. 迁移 Labs：将静态 LABS 数据和展示组件放入 `features/labs`。
4. 决策并推进轻量后端模块的统一响应。
5. 直接在 `backend/main.py` 注册 `modules/*/router.py`，逐步删除 `backend/api/*_routes.py` 兼容层。

## 当前完成度判定

| 层级 | 判定 |
|---|---|
| 样板模块 | Radar、Graph 已完成，可作为后续迁移模板 |
| 后端 API 模块 | 大部分已迁入 `modules`，剩余主要是注册入口和响应策略 |
| 前端业务模块 | 核心展示模块完成较多，管理类和首页聚合仍待收敛 |
| 测试基础 | 已可支撑继续小步迁移 |
| 风险状态 | 中等，主要风险来自前端页面剩余直连 SQLite 和响应契约不完全统一 |

结论：可以继续按“小步迁移 + 每批服务层测试 + 保持旧契约”的策略推进。下一批最推荐迁移 Home，其次是 CMS。
## 2026-07-08 follow-up migration update

- Home is now split into `features/home/components/HomePageView.tsx`, `features/home/data/homeContent.ts`, and `features/home/server/homeService.ts`; `app/page.tsx` only loads stats and renders the feature view.
- CMS is now split into feature views for dashboard, editor, generation, and publishing; `app/cms/*` pages only route, load data, or pass route params.
- Labs metadata is now owned by `features/labs/data/labs.ts`; `app/labs/*` no longer embeds lab data.
- Learn progress writes now go through `features/progress/api/progressApi.ts`; client components no longer call `fetch('/api/progress')` directly.
- Backend `main.py` now registers `modules/*/router.py` directly; the old `backend/api/*_routes.py` compatibility layer has been removed.
- Remaining follow-up: decide whether legacy direct-response endpoints should be wrapped with `success_response/error_response` module by module.
