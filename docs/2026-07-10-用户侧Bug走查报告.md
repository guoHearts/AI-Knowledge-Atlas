## 二、问题清单

### P0-1｜知识图谱页搜索结果完全不可见，搜索功能不可用

- **页面**: `/graph`
- **复现步骤**: 打开图谱页 → 在顶部搜索框输入任意关键词（如 "RAG"）
- **现象**: 无障碍树（accessibility tree）中能看到 5 条正确的搜索结果（含节点名称、简介、分类），下拉框的 DOM 颜色样式（深色背景、浅色文字、`z-index: 50`）也完全正确；但截图显示**下拉框区域是空白的**，被图谱画布盖住，用户看不到、也点不到任何搜索结果。
- **根因**:
  - `frontend/src/features/graph/components/GraphToolbar.tsx:25` 的工具栏容器使用了 `backdrop-blur-xl`（`backdrop-filter`），这会为该元素创建新的 **CSS 层叠上下文（stacking context）**，且该容器本身没有设置 `z-index`。
  - `frontend/src/features/graph/components/GraphWorkspaceView.tsx:37` 中，图谱画布的包裹容器在 DOM 顺序上排在工具栏之后，两者都是 `z-index: auto`，于是画布作为后一个兄弟元素在层叠顺序上盖过了工具栏整个层叠上下文——连带把工具栏内部 `SearchBar.tsx:66` 设置的 `z-50` 下拉框也一起遮住了。
- **影响**: 图谱页的核心入口（搜索）对真实用户完全失效，只能通过点击/双击节点浏览，无法快速定位目标节点。
- **建议修复方向**: 给 `GraphToolbar` 外层容器显式设置更高的 `z-index`（并配合 `position: relative`），或将 `SearchBar` 的下拉框改为 `position: fixed` / 挂载到更高层级（如 Portal），避免被兄弟层叠上下文盖住。

### P1-1｜CMS "Course editor" 入口点击后 404

- **页面**: `/cms` → 点击 "Course editor" 卡片
- **复现步骤**: 打开 `/cms` → 点击顶部三张卡片中的 "Course editor"
- **现象**: 跳转到 `/cms/editor`，返回 **HTTP 404**，控制台报 `Failed to load resource: 404`。而页面下方 "Track overview" 里的 "Edit" 链接（跳转到 `/cms/editor/agent-engineer`）是正常的。
- **根因**:
  - `frontend/src/features/cms/components/CmsDashboardView.tsx:20` 硬编码了 `href: '/cms/editor'`。
  - 实际路由只有 `frontend/src/app/cms/editor/[...slug]/page.tsx`，是**必填参数**的 catch-all 路由（非 `[[...slug]]` 可选形式），不带任何 slug 时无法匹配，因此 404。
- **影响**: 首页三大功能入口之一直接不可用。
- **建议修复方向**: 要么把卡片链接改成指向默认/第一个 track（如 `/cms/editor/agent-engineer`），要么把 catch-all 路由改成可选 `[[...slug]]` 并在无参数时展示 track 选择列表。

### P2-1｜雷达详情页分类徽章大小写错误（"Mcp" 而非 "MCP"）

- **页面**: `/radar/mcp-security-boundary-2026-07`
- **现象**: 页面顶部分类徽章显示为 **"Mcp"**，而不是正确的 "MCP"（列表页 `/radar` 上同一条目显示的是完整的 "Model Context Protocol"，图谱关联区域也正确显示 "MCP"）。
- **根因**: `frontend/src/features/radar/components/RadarItemDetailView.tsx:5-6`：
  ```ts
  function formatCategory(category: string) {
    return category.replace(/-/g, ' ').replace(/\b\w/g, (letter) => letter.toUpperCase());
  }
  ```
  直接对原始分类 id（`mcp`）做逐词首字母大写，而不是像列表卡片 `RadarItemCard.tsx:58` 那样查 `categories` 数组取正规显示名。
- **影响**: 视觉上的专业度问题，用户会看到不规范的缩写大小写。
- **建议修复方向**: 详情页复用 `RadarItemCard.tsx` 里 `categories.find(...).name` 的查表逻辑，而不是用正则推断。

### P2-2｜首页"学习地图"卡片文字被徽章遮挡（长文案换行时）

- **页面**: 首页 `/`，"学习地图 / Learning Map" 区块，"Product systems / AI 产品模式" 卡片
- **复现步骤**: 桌面宽度下打开首页，滚动到"学习地图"区块查看第 4 张卡片
- **现象**: 卡片描述文字换行到第 4 行时，底部绝对定位的 "Module 12" 徽章会盖住最后一行文字（英文文案下表现为 "...deployment **w**orkflows." 中的 "w" 被徽章挡住）。
- **根因**: `frontend/src/features/home/components/HomePageView.tsx:137-147`。卡片容器只设置了 `min-h-[250px]`（最小高度，非固定高度），而徽章用 `absolute bottom-5 left-5` 定位、没有为其预留底部内边距；当描述文本足够长换到 4 行时，正文自然高度会顶到容器底部，与绝对定位的徽章重叠。
- **建议修复方向**: 给 `<p>` 描述文字增加底部 padding（预留出徽章高度），或把徽章从绝对定位改为正常文档流（放在描述文字下方）。

### P2-3｜界面语言切换到中文后，部分内容仍固定显示英文

- **涉及页面**: 首页"学习地图 / 近期工作"区块、雷达详情页正文（为什么重要/适合谁/不适合）、CMS 管理后台整体文案、Labs 实验页大部分内容
- **现象**: 点击顶部"切换语言"按钮从 EN 切回中文后，导航栏、页面标题等 UI 文案都正确变成中文，但上述区块的**实际内容文本**仍然是英文，造成中英文混排。
- **根因（已确认的一处）**: 首页"学习地图"与"近期工作"两个区块的数据来自后端接口 `/learning/home/content`：
  - `backend/modules/learning/service.py:56-60`：
    ```python
    def get_home_content(self) -> dict[str, Any]:
        return {
            "roadmap": catalog.ROADMAP,
            "nextSteps": catalog.NEXT_STEPS,
        }
    ```
  - 该接口返回硬编码的英文 `catalog.ROADMAP` / `catalog.NEXT_STEPS`，**不接受也不处理 locale 参数**，无论前端选择哪种语言，返回内容都是同一份英文数据。
  - 前端调用处 `frontend/src/features/home/api/homeApi.ts:33-35` 的 `getHomeContent()` 同样没有传任何语言参数。
  - 雷达 / CMS / Labs 的英文内容更可能是编辑内容尚未产出中文版本（内容缺口），而非同一段代码逻辑问题，但对用户呈现的割裂体验是一致的。
- **建议修复方向**: 短期给 `catalog.ROADMAP` / `catalog.NEXT_STEPS` 补充中文版本并让接口按 `Accept-Language` 或显式 locale 参数返回对应文案；中长期看是否需要为雷达/Labs/CMS 内容建立统一的中英双语内容规范（可参考 `docs/content-standards.md`）。

### P3-1｜学习路线页"学完你将能够"板块内容为空

- **页面**: `/learn/agent-engineer`（页面底部）
- **现象**: "🎓 学完你将能够" 区块下只显示了 "🏆 毕业项目" 标签，技能列表和毕业项目描述均为空白，没有任何实际内容。
- **根因**: `frontend/src/app/learn/[track]/page.tsx:150-164` 渲染的是 `track.outcomeSkills`（数组）与 `track.outcomeProject`（字符串），对应数据库字段 `outcome_skills` / `outcome_project`（见 `docker/postgres/init/001_learning_schema.sql`）。当前 `agent-engineer` 这条 track 记录里这两个字段为空，属于**数据缺失**而非代码渲染问题。
- **建议修复方向**: 为 `agent-engineer` track 补全 `outcome_skills` 和 `outcome_project` 数据；同时建议前端在数据为空时隐藏整个区块或显示占位提示，避免出现"标题在、内容空"的半成品观感。
