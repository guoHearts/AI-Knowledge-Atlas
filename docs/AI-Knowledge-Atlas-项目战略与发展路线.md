# AI-Knowledge-Atlas 项目战略与发展路线

> 面向 AI 开发者的持续更新、来源可追溯、代码可运行的 AI Engineering 决策与验证平台。

## 1. 文档目的

本文档用于统一 AI-Knowledge-Atlas 的项目定位、目标用户、产品边界、发展方向、阶段目标和执行顺序。

后续新增页面、课程、实验、采集器和图谱能力，都应先回答两个问题：

1. 是否帮助 AI 开发者做出更可靠的技术判断？
2. 是否强化“来源可追溯、代码可运行、时间可核验”的项目可信度？

无法回答上述问题的需求，原则上不进入当前阶段的开发计划。

---

## 2. 当前阶段判断

AI-Knowledge-Atlas 已经具备可运行的工程基础，包括：

- Next.js 前端；
- FastAPI 后端；
- Neo4j 技术知识图谱；
- SQLite 学习与进度数据；
- Agent 学习路线和课程结构；
- CMS、数据采集和内容管理雏形；
- 端到端、API、数据库和性能测试基础。

当前主要问题不是“功能太少”，而是：

- 项目定位仍然偏散；
- 用户进入仓库后，难以在 30 秒内理解核心价值；
- 内容数量大于内容可信度；
- 知识图谱偏展示，尚未充分参与技术决策；
- 可运行实验不足，无法形成强信任；
- 缺少稳定、可持续的内容更新节奏；
- 项目目前更像完成度较高的个人工程作品，还没有形成外部用户持续使用的理由。

因此，下一阶段不再以增加课程数量、节点数量和页面数量为主要目标，而是建立一个清晰、可信、可运行的核心产品闭环。

## 3. 项目愿景、使命与核心定位

### 3.1 愿景

成为中文 AI 开发者在了解新技术、进行技术选型和快速验证方案时，优先查阅的开放式 AI Engineering 决策库。

### 3.2 使命

把快速变化的新模型、新框架、新协议、新论文和工程实践，转化为开发者可以理解、比较、运行和验证的技术资产。

### 3.3 一句话定位

> AI-Knowledge-Atlas 是一个面向 AI 开发者的持续更新技术雷达，通过可信技术解读、选型地图、可运行实验和知识图谱，帮助开发者判断一项技术是否值得关注、何时使用以及如何验证。

### 3.4 核心承诺

项目不只回答“发生了什么”，还必须回答：

- 为什么重要；
- 解决了什么问题；
- 与已有方案相比发生了什么变化；
- 适合什么场景；
- 不适合什么场景；
- 当前是否适合生产环境；
- 如何用最小成本运行验证；
- 与其他模型、框架、协议和工程模式有什么关系。

## 4. 目标用户

### 4.1 核心用户

- AI 应用开发者；
- Agent、RAG、MCP 开发者；
- 后端和全栈开发者；
- AI 平台工程师；
- 正在转向 AI Engineering 的传统软件开发者。

### 4.2 次级用户

- 技术负责人和架构师；
- AI 产品经理；
- 技术内容创作者；
- 高校学生和研究人员。

### 4.3 用户核心任务

用户来到项目，不是为了浏览一份静态百科，而是为了完成以下任务：

1. 快速判断最近哪些 AI 技术值得关注；
2. 在多个模型、SDK、框架和协议之间进行选型；
3. 判断一项方案是否已经适合真实项目；
4. 通过可运行实验验证官方宣传或技术结论；
5. 理解一项技术的上下游依赖、替代方案、风险和演进路径；
6. 将零散的新知识沉淀为系统学习路线。

## 5. 核心价值与差异化

### 5.1 核心价值链

```text
发现技术变化
    ↓
筛选是否值得关注
    ↓
形成明确技术判断
    ↓
与现有方案进行对比
    ↓
通过可运行实验验证
    ↓
连接到知识图谱
    ↓
沉淀为长期学习内容
```

### 5.2 核心差异化

项目不以“内容多”作为竞争优势，而以以下三点建立信任：

1. **来源可追溯**：技术结论优先引用官方文档、官方 GitHub、官方博客和论文；
2. **代码可运行**：关键结论尽量附带最小实验、测试结果和预期输出；
3. **时间可核验**：所有内容标记创建、更新、最后验证时间和依赖版本。

### 5.3 不参与的竞争

项目不追求：

- 成为覆盖所有 AI 概念的大百科；
- 成为高频搬运新闻的资讯聚合站；
- 依靠 AI 批量生成大量低质量课程；
- 用无法解释的评分制造“专业感”；
- 在早期建设复杂的社交、推荐和付费系统；
- 与官方文档竞争完整 API 参考手册。

## 6. 产品结构

产品长期保持五个一级模块：

```text
AI-Knowledge-Atlas
├── Radar       AI 技术雷达
├── Compare     技术选型与对比
├── Labs        可运行实验
├── Atlas       技术知识图谱
└── Learn       系统学习路线
```

### 6.1 Radar：发现入口

Radar 负责发现和筛选值得关注的技术变化。

每条内容必须包含：

- 一句话结论；
- 为什么重要；
- 技术成熟度；
- 推荐用户；
- 不适用场景；
- 官方来源；
- 最后验证日期；
- 相关 Compare、Lab、图谱节点或课程。

每周只发布 5～10 条高价值内容，不追求数量。

### 6.2 Compare：技术决策

Compare 负责回答真实的技术选型问题。

每篇对比至少说明：

- 核心设计差异；
- 功能与工程能力矩阵；
- 典型适用和不适用场景；
- 学习、迁移和部署成本；
- 模型兼容性、状态管理、可观测性和安全边界；
- 最终建议与决策树；
- 对应可运行实验。

### 6.3 Labs：信任核心

Labs 是项目最重要的信任资产。

每个 Verified Lab 必须具备：

- 清晰的业务目标；
- 固定依赖版本；
- 一条命令启动；
- 环境变量示例；
- 预期输入与输出；
- 测试和 CI；
- 常见失败场景；
- 安全注意事项；
- 预计运行成本；
- 最后验证日期；
- 相关官方来源和图谱节点。

首批重点实验：

1. Production Agent；
2. Hybrid RAG Evaluation；
3. Secure MCP Server。

### 6.4 Atlas：关系与决策辅助

知识图谱不再只是展示节点，而应回答：

- 一项技术依赖什么；
- 它替代了什么；
- 有哪些备选方案；
- 适合哪些场景；
- 有哪些已知风险；
- 被哪些实验验证；
- 对应哪些课程和文章。

### 6.5 Learn：长期知识沉淀

Learn 负责承载相对稳定、长期有效的知识，不承担所有最新新闻。

建议优先维护以下路线：

- AI Engineering Foundation；
- Agent Engineering；
- RAG Engineering；
- MCP 与工具生态；
- LLM Evaluation；
- Production AI / LLMOps；
- AI Security。

课程只有在基础内容稳定且有人使用时才继续扩张。

## 7. 内容可信度体系

### 7.1 内容状态

所有 Radar、Compare、Lab 和课程统一使用：

| 状态 | 定义 |
|---|---|
| `Verified` | 已完成来源、版本、代码或结论验证 |
| `Stale` | 依赖、接口、模型或结论可能已经发生变化 |
| `Draft` | 尚未完成验证，不应作为正式结论传播 |
| `Deprecated` | 已废弃、不再推荐或仅保留历史参考 |

### 7.2 Verified 准入规则

内容只有满足以下条件才能标记为 `Verified`：

- 引用至少一个官方一手来源；
- 标注依赖和模型版本；
- 标注最后验证日期；
- 关键代码可以编译或运行；
- 记录预期输出；
- 说明适用和不适用场景；
- 说明已知限制与风险；
- 结论不夸大实验性能力；
- 相关 Lab 存在时，CI 必须通过。

### 7.3 自动失效规则

```text
90 天未验证          → Needs Review
依赖主版本升级        → Stale
官方 API 废弃         → Deprecated
Lab CI 失败           → 移除 Verified
来源链接长期失效       → Stale
```

### 7.4 发布流程

```text
发现选题
  ↓
核查官方来源
  ↓
形成初步结论
  ↓
实现最小实验
  ↓
本地运行与测试
  ↓
人工复核
  ↓
标记 Verified
  ↓
发布并进入过期监控
```

## 8. 发展方向

### 8.1 第一发展方向：可信技术决策

项目的第一优先级不是资讯速度，而是决策质量。

重点内容应围绕高频问题：

- Agent 框架如何选；
- MCP、Function Calling 和 REST API 如何组合；
- 向量检索、混合检索、GraphRAG 如何选；
- 闭源、开源和自托管模型如何选；
- Workflow 与 Autonomous Agent 如何选；
- 新模型或 SDK 是否值得迁移；
- 一项技术距离生产可用还缺什么。

### 8.2 第二发展方向：可复现工程实验

项目逐步从“讲概念”转向“给出最小可复现工程”。

实验不追求规模，而追求：

- 可运行；
- 可验证；
- 可维护；
- 可复用；
- 可以说明真实工程边界。

### 8.3 第三发展方向：技术关系图谱

图谱应由静态展示升级为决策基础设施，为搜索、推荐、对比和课程关联提供统一数据关系。

### 8.4 第四发展方向：半自动内容流水线

自动化只负责：

- 发现更新；
- 去重和分类；
- 提取官方来源；
- 生成候选摘要；
- 检查版本和链接；
- 触发过期提醒。

人工负责：

- 判断是否值得发布；
- 形成最终结论；
- 验证代码和风险；
- 决定成熟度和推荐场景。

项目不允许自动采集内容未经审核直接标记为 Verified。

### 8.5 第五发展方向：社区共建

当内容规则和实验模板稳定后，再逐步引入：

- Content Correction；
- New Technology Request；
- Lab Request；
- Good First Issue；
- 外部技术对比和实验贡献；
- 中英文协作。

## 9. 项目边界与优先级

### 9.1 当前必须做

- 修复错误或过时内容；
- 重写 README 和首页定位；
- 建立内容状态和版本验证机制；
- 完成一个高质量 Verified Lab；
- 建立第一期 Radar；
- 建立 Compare 模板；
- 让图谱连接 Radar、Compare、Lab 和 Learn；
- 增加基础开源治理文件和 CI。

### 9.2 当前暂缓

- 复杂用户账号体系；
- 评论、私信和社区动态；
- 个性化推荐算法；
- 大量新课程；
- 移动 App；
- 复杂 CMS 工作流；
- 商业订阅和支付；
- 大规模实时爬虫；
- 没有真实用户需求支撑的企业功能。

### 9.3 明确停止

- 为了展示规模而增加低价值图谱节点；
- 未验证内容标记为 Verified；
- 使用不存在或过时的 SDK 接口；
- 直接发布 AI 自动生成的技术结论；
- 使用无法解释的成熟度分数；
- 同时开发多个没有标杆质量的 Lab。

## 10. 分阶段路线图

本路线按个人开源项目、时间有限、月度工具预算约 100 元的条件设计。优先保证持续性，不依赖大规模模型调用或全职团队。

### 阶段 0：可信度清理期

**时间：第 1～2 周**

目标：清除影响信任的错误和过期内容。

交付物：

- 全量检查现有 `Verified` 内容；
- 不符合要求的内容降级为 `Draft` 或 `Stale`；
- 统一 Frontmatter；
- 增加内容验证脚本；
- 修复 README 中不准确的定位和数据；
- 增加 LICENSE、CONTRIBUTING、SECURITY、ROADMAP 和 Issue 模板。

验收标准：

- 现有正式内容不包含已知错误接口；
- 所有正式内容有来源和最后验证日期；
- `Verified` 状态可以被解释和审计。

### 阶段 1：核心产品成立

**时间：第 3～6 周**

目标：让外部用户第一次感受到项目的独特价值。

交付物：

- 新 README 第一屏；
- 新首页第一屏；
- 第一份 AI Engineering Radar；
- 第一篇高质量技术选型；
- 第一个 Verified Lab；
- Radar、Compare、Lab、Atlas、Learn 之间建立跳转关系；
- 发布 `v0.1.0 — Positioning Reset`。

验收标准：

- 用户 30 秒内理解项目；
- 用户可以在 10 分钟内运行一个真实实验；
- 核心结论有官方来源和版本信息；
- 项目至少有一个可单独传播的成果。

### 阶段 2：形成持续更新能力

**时间：第 2～3 个月**

目标：从一次性作品变成持续更新的产品。

交付物：

- 每周一期 Radar；
- 每两到四周一个 Lab；
- 每月至少两篇高质量 Compare；
- 自动检查链接、Frontmatter、依赖版本和过期状态；
- 图谱支持查看替代方案、适用场景、风险和验证实验；
- 建立公开 Roadmap 和 Changelog。

验收标准：

- 连续更新至少 8 周；
- 核心内容验证率达到 100%；
- 至少 3 个 Verified Labs；
- 开始出现外部 Issue、Fork、PR 或真实使用反馈。

### 阶段 3：验证外部用户价值

**时间：第 4～6 个月**

目标：证明项目不只是作者个人使用。

交付物：

- 建立最小匿名使用数据；
- 收集 Lab 运行、内容收藏、回访和外部贡献数据；
- 优化高访问技术专题；
- 完成英文 README 和核心内容翻译；
- 建立轻量 Newsletter 或更新订阅；
- 发布 `v0.5.0 — Verified Engineering Atlas`。

验收标准：

- 有稳定回访用户；
- 有外部贡献者；
- 有内容被开发者引用或用于真实项目；
- 能明确说出用户最依赖的前三项能力。

### 阶段 4：平台化与生态化

**时间：第 7～12 个月**

只有阶段 3 的用户价值成立后，才进入本阶段。

可能方向：

- 公开内容 API；
- 技术更新订阅；
- 社区维护者机制；
- 可复用 Lab 模板；
- 企业内部技术雷达部署；
- Sponsor、咨询、培训或企业定制。

项目不预设一定商业化，商业能力必须建立在可信内容和真实使用基础上。

## 12. 关键指标

### 12.1 北极星指标

> **每月被真实运行或用于技术决策的 Verified 内容数量。**

该指标比单纯统计页面访问和 Star 更接近项目价值。

### 12.2 产品指标

- Verified 内容数量；
- Verified Labs 数量；
- Lab 成功运行次数；
- Radar 到 Compare/Lab 的点击率；
- 内容回访率；
- 过期内容比例；
- 内容平均验证周期；
- 搜索后进入有效内容的比例。

### 12.3 开源指标

- 外部 Issue 数量；
- 外部 PR 数量；
- 外部贡献者数量；
- Fork 和 Star；
- Release 下载和仓库 Clone；
- 被其他文章、项目或社区引用的次数。

### 12.4 第一阶段目标

未来 90 天建议目标：

- 3 个 Verified Labs；
- 8～12 期 Radar 或至少连续更新 8 周；
- 6 篇高质量 Compare；
- 100% 核心内容有官方来源和验证日期；
- 0 个已知无法运行但标记为 Verified 的示例；
- 10 个 Good First Issues；
- 获得第一批真实外部反馈或贡献。

Star 不作为唯一目标，也不设脱离内容质量的强制数字。

## 13. 技术与成本原则

### 13.1 技术原则

- 优先复用现有 Next.js、FastAPI、Neo4j 和 SQLite 技术栈；
- 不因架构兴趣频繁更换框架；
- 先通过 Markdown Frontmatter 落地内容模型，再决定是否迁移数据库；
- CI 优先验证关键内容和 Lab，不追求复杂流水线；
- 只有真实访问量和数据规模出现后，才考虑缓存、搜索集群和微服务拆分。

### 13.2 AI 使用原则

在月度工具预算约 100 元的条件下：

- 使用低成本模型完成采集去重、分类、格式化和初稿；
- 高价值技术判断由人工复核；
- 只有复杂源码分析和难题才使用高成本模型；
- 自动化任务必须限制 Token、并发和执行频率；
- 不把模型生成内容直接标记为 Verified；
- 优先使用本地脚本、CI 和确定性校验代替重复模型调用。

### 13.3 维护原则

项目按照“一个稳定主线 + 一个当前标杆任务”推进。

任意时间最多同时维护：

- 一个核心产品迭代；
- 一个标杆 Lab；
- 一期 Radar；
- 一篇 Compare。

避免多线并行导致大量半成品。

## 15. 风险与应对

### 风险 1：项目范围过大

**表现**：同时做资讯、课程、图谱、CMS、社区和企业功能。

**应对**：未来 90 天只围绕 Radar、Compare、Labs 和可信度建设，Atlas 与 Learn 只做关联改造。

### 风险 2：内容快速过期

**表现**：SDK、模型和接口更新后，旧内容仍被标记为 Verified。

**应对**：建立最后验证日期、依赖监控和自动 Stale 规则。

### 风险 3：AI 批量生成降低可信度

**表现**：内容看似完整，但版本、接口或结论错误。

**应对**：AI 只负责候选内容，Verified 必须经过人工和运行验证。

### 风险 4：个人维护精力不足

**表现**：多个模块同时开发，更新节奏中断。

**应对**：减少并行任务；每月只交付一个最重要的完整成果；优先维护已有高价值资产。

### 风险 5：项目只有 Star，没有真实使用

**表现**：传播数据增长，但没有 Lab 运行、回访和贡献。

**应对**：将 Lab 运行、回访、引用和外部反馈作为核心指标。

### 风险 6：知识图谱成为装饰

**表现**：图谱节点很多，但无法帮助选型。

**应对**：所有新增关系必须服务于依赖、替代、适用场景、风险或验证链路。

## 17. 项目决策原则

后续需求按以下顺序判断：

1. 是否解决目标用户的真实问题；
2. 是否增强项目可信度；
3. 是否能够形成可运行或可验证结果；
4. 是否能复用现有架构；
5. 是否能够由个人长期维护；
6. 是否值得占用当前有限预算和时间。

任何功能即使技术上有趣，只要不满足前三项，就不应成为当前优先任务。

## 18. 最终目标状态

AI-Knowledge-Atlas 的最终价值不在于拥有多少页面、课程和图谱节点，而在于形成如下稳定闭环：

```text
自动发现变化
    ↓
人工与 AI 协同筛选
    ↓
形成可信技术雷达
    ↓

    ↓
制作可运行验证实验
    ↓
连接技术知识图谱
    ↓
沉淀为长期学习内容
    ↓
通过社区反馈持续修正
```

用户对项目形成的最终认知应当是：

> 当我想了解 AI 开发领域最近发生了什么、某项技术是否值得学习、应该如何选型，以及怎样低成本运行验证时，我会打开 AI-Knowledge-Atlas。
