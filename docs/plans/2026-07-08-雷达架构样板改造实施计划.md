# 雷达架构样板改造实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use inline execution task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将 Radar 改造成前端 `features/radar` + 后端 `modules/radar` 的架构样板，并建立统一响应、请求封装和最小测试保护。

**Architecture:** 前端 `app` 只保留路由入口，Radar 业务组件、hooks、API 和类型全部进入 `features/radar`。后端 Radar 拆成 `router/service/repository/schemas`，统一通过 `common/response` 和 `common/errors` 输出稳定 API 响应。Next.js 服务端请求使用 `BACKEND_INTERNAL_URL`，浏览器侧使用相对 `/api` 路径。

**Tech Stack:** Next.js 16、React 19、TypeScript strict、FastAPI、Pydantic v2、pytest、pnpm。

## Global Constraints

- 文件命名文档使用日期 + 中文说明。
- 不覆盖用户已有的 `Make.ps1`、`start.ps1` 和新增架构文档改动。
- 不新增 Radar 业务内容，不引入新数据库。
- 新业务代码进入 `frontend/src/features/radar` 和 `backend/modules/radar`。
- 前端组件不直接 `fetch`。
- 后端 router 不保存 sample data，不写筛选和排序业务。
- API 成功响应包含 `success`、`data`、`message`、`requestId`。
- API 失败响应包含 `success`、`error`、`requestId`。
- 服务端后端地址通过 `BACKEND_INTERNAL_URL` 获取。
