# 图谱架构样板改造设计

## 背景

Radar 已完成第一批架构样板改造，形成了 `features/radar`、`modules/radar`、统一 request、统一响应和服务端内部后端地址的模板。

Graph 是项目第二个核心模块，也是当前架构债较集中的模块：

- `frontend/src/app/graph/page.tsx` 同时处理路由、请求、状态、画布交互、筛选、图例和详情面板。
- `frontend/src/lib/api.ts` 承载 Graph 业务 API，并直接使用 `NEXT_PUBLIC_API_URL || http://localhost:8000` 发起 `fetch`。
- `backend/api/graph_routes.py` 直接在路由中处理业务分支，并通过 `from main import app` 读取 Neo4j driver。
- `backend/api/chat_routes.py` 也通过 `from main import app` 获取 repository，和 Graph 依赖方式重复。

根据 `docs/agent需遵循架构约定.md`，第二批改造应继续沿用 Radar 的小步样板策略，而不是一次性迁移 Learn、Labs、CMS。

## 目标

本阶段目标是把 Graph 浏览、搜索、详情、展开、统计和同步入口改造成可复用架构样板：

1. 前端新增 `frontend/src/features/graph`，将 Graph 页面业务逻辑从 `app/graph/page.tsx` 下沉。
2. 将 Graph API 从 `frontend/src/lib/api.ts` 迁移到 `features/graph/api/graphApi.ts`，统一走 `lib/request.ts`。
3. 后端新增 `backend/modules/graph`，将 Graph HTTP 路由、service、repository provider 和 schemas 分层。
4. 移除 Graph 路由中的 `from main import app` 直接依赖，改用明确的 FastAPI `Request` dependency。
5. 保持现有 Graph 浏览行为不变：加载节点/边、搜索、点击详情、双击展开、focus 参数、筛选和图例。
6. 增加最小测试，覆盖 Graph service 行为和前端 Graph API 路径。

## 非目标

本阶段不做以下事情：

- 不改 Neo4j 图数据结构、索引或 Cypher 查询语义。
- 不重写 D3/Canvas 布局算法。
- 不重写 RAG Chat 生成逻辑，不改变 prompt、LLM 调用或回答格式。
- 不迁移 Learn、Labs、CMS。
- 不引入新状态库。
- 不引入新后端数据库或 ORM。
- 不改用户未提交的 `Make.ps1`、`start.ps1` 和新增架构文档。

## 前端设计

### 目录结构

新增 Graph feature：

```txt
frontend/src/features/graph/
├─ api/
│  └─ graphApi.ts
├─ components/
│  ├─ GraphWorkspaceView.tsx
│  ├─ GraphToolbar.tsx
│  ├─ GraphLegend.tsx
│  └─ GraphRelationsBadgeList.tsx
├─ hooks/
│  └─ useGraphWorkspace.ts
├─ types/
│  └─ graph.types.ts
└─ index.ts
```

`frontend/src/app/graph/page.tsx` 只保留：

- `Suspense` 边界；
- `useSearchParams` 读取 `focus`；
- 渲染 `GraphWorkspaceView`。

现有全局图谱组件暂不强制迁移：

- `frontend/src/components/graph/GraphCanvas.tsx`
- `frontend/src/components/graph/NodeDetail.tsx`
- `frontend/src/components/graph/SearchBar.tsx`
- `frontend/src/components/graph/FilterPanel.tsx`

这些组件短期仍作为图谱专属 UI 组件被 `features/graph` 使用。后续如果继续严格收敛，可以再单独迁入 `features/graph/components`。本阶段不做这一步，避免 D3 画布改动扩大风险。

### 数据流

```txt
app/graph/page.tsx
  ↓
features/graph/components/GraphWorkspaceView.tsx
  ↓
features/graph/hooks/useGraphWorkspace.ts
  ↓
features/graph/api/graphApi.ts
  ↓
lib/request.ts
  ↓
FastAPI /graph endpoints
```

### API 封装

`features/graph/api/graphApi.ts` 暴露：

- `listGraphNodes(params?: { type?: NodeType; limit?: number })`
- `listGraphEdges(limit?: number)`
- `searchGraphNodes(query: string)`
- `getGraphNodeDetail(nodeType: string, nodeId: string)`
- `getGraphSubgraph(ids: string[], depth?: number)`
- `getGraphStats()`

迁移后 `frontend/src/lib/api.ts` 不再承载 Graph 业务 API。为了降低风险，可以删除该文件，或只保留兼容 re-export；若没有其他调用方，优先删除。

### 状态和交互

`useGraphWorkspace` 负责：

- 初始加载节点和边；
- 搜索选择后打开详情；
- 点击节点打开详情；
- 双击节点拉取一度子图并合并去重；
- focus 参数触发初始详情打开；
- 筛选类型增删；
- 面板开关、highlight、focus 状态。

`GraphWorkspaceView` 负责组合：

- toolbar；
- canvas；
- legend；
- relation badges；
- detail panel。

## 后端设计

### 目录结构

新增 Graph 模块：

```txt
backend/modules/graph/
├─ dependencies.py
├─ router.py
├─ service.py
└─ schemas.py
```

暂不迁移 `backend/models/repository.py` 内的 `GraphRepository`，因为它已经是数据访问层，且被 scheduler、chat、seed 脚本复用。本阶段只建立模块边界，不移动共享 repository，避免破坏其他调用方。

### 分层职责

`dependencies.py`：

- 从 FastAPI `Request` 读取 `app.state.neo4j_driver`。
- 构造 `GraphRepository`。
- 从 `app.state.pipeline` 读取可选 pipeline。
- 替代 `from main import app`。

`service.py`：

- 编排 Graph 查询业务。
- 将“不存在节点”“pipeline 未初始化”等错误转成 `AppError`。
- 不直接处理 HTTP。

`router.py`：

- 读取 HTTP query/path 参数。
- 调用 service。
- 返回统一响应。
- 不直接访问 `app.state`。

`schemas.py`：

- 定义前端所需列表包装结构，如 `GraphNodesData`、`GraphEdgesData`。
- 复用现有 `models.graph` 的 `GraphNode`、`GraphEdge`、`NodeDetail`、`Subgraph`。

### Chat 处理

本阶段不重写 `chat_routes.py` 的业务逻辑，但会允许它复用 `modules/graph/dependencies.py` 中的 `get_graph_repository`，从而去掉重复的 `from main import app`。Chat 响应格式暂时保持原样，避免影响聊天前端或未来 RAG 改造。

## API 契约

Graph 浏览接口改为统一响应：

```json
{
  "success": true,
  "data": {},
  "message": "ok",
  "requestId": "req_xxx"
}
```

列表节点：

```txt
GET /graph/nodes?node_type=Technology&limit=100
```

返回：

```json
{
  "success": true,
  "data": {
    "items": []
  },
  "message": "ok",
  "requestId": "req_xxx"
}
```

边列表：

```txt
GET /graph/edges?limit=500
```

搜索：

```txt
GET /graph/nodes/search?q=rag&limit=20
```

详情：

```txt
GET /graph/nodes/{nodeType}/{nodeId}
```

子图：

```txt
GET /graph/subgraph?ids=rag&depth=1
```

统计：

```txt
GET /graph/stats
```

同步：

```txt
POST /graph/admin/sync
```

第一阶段 Graph 错误码：

- `GRAPH_NODE_NOT_FOUND`
- `GRAPH_PIPELINE_UNAVAILABLE`
- `VALIDATION_ERROR`
- `INTERNAL_SERVER_ERROR`

## 兼容策略

前端 `lib/request.ts` 已兼容旧直接响应和新统一响应。因此后端 Graph 可以直接切到统一响应，前端迁移到 `graphApi.ts` 后会正常解包。

如果仍有未迁移调用方直接请求 `/graph/*`，它们需要适配新响应结构。本阶段只保证项目内前端调用方完成迁移。

## 风险与缓解

### 风险：Graph 画布交互回归

缓解：不改 `GraphCanvas` 的 D3 逻辑，只迁移调用边界和状态组织；构建通过后再做页面级手动或浏览器验证。

### 风险：统一响应影响未迁移调用方

缓解：先用 `rg` 确认项目内 `/graph/*` 调用方，确保前端迁移完成；`lib/request.ts` 已兼容旧响应，但直接 `fetch` 调用方必须消除。

### 风险：移动 repository 影响 scheduler/chat/seed

缓解：本阶段不移动 `models.repository.GraphRepository`，只在 `modules/graph` 包装 service 和 dependencies。

### 风险：Chat 重构扩大范围

缓解：Chat 只复用 dependency，不改 RAG 业务、不改响应格式。

## 完成标准

- `frontend/src/app/graph/page.tsx` 只作为路由入口和 Suspense 包装。
- Graph 页面业务状态集中在 `features/graph/hooks/useGraphWorkspace.ts`。
- Graph API 封装集中在 `features/graph/api/graphApi.ts`。
- Graph 前端业务代码不再依赖 `frontend/src/lib/api.ts`。
- `backend/api/graph_routes.py` 不再包含业务实现，只兼容导出新 router。
- Graph 后端通过 `modules/graph/service.py` 编排业务。
- Graph 路由和 Chat 路由不再使用 `from main import app` 获取 repository。
- 新增测试覆盖 Graph service 和 Graph API。
- `pytest`、`pnpm test`、`pnpm typecheck`、`pnpm lint`、`pnpm build` 通过。
