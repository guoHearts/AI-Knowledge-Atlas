# 通用前后端架构规范文档

版本：v1.0
适用范围：中小型 Web 应用、管理后台、SaaS 系统、内容系统、AI 工具类项目
推荐技术形态：Next.js 前端 + 后端 API 服务 + PostgreSQL/MySQL + Docker Compose

---

## 1. 文档目标

本文档用于统一项目的前后端架构设计、目录组织、模块边界、接口约定、数据流、错误处理、环境配置和协作规范。

本文档关注的是工程化架构，不是单个功能的实现细节。核心目标是：

1. 降低项目复杂度。
2. 明确前端、后端、数据库的职责边界。
3. 统一代码目录、命名、接口、错误码和数据结构。
4. 提高多人协作、后期维护和功能扩展的稳定性。
5. 避免业务代码无序堆积，形成可持续演进的项目结构。

---

## 2. 总体架构原则

### 2.1 分层清晰

系统需要按照职责分层，禁止不同层之间职责混杂。

推荐分层：

```txt
用户界面层
  ↓
前端业务层
  ↓
前端请求层
  ↓
后端接口层
  ↓
后端业务层
  ↓
数据访问层
  ↓
数据库
```

每一层只处理自己该处理的问题。

例如：

* 页面不直接操作数据库。
* 组件不直接拼接后端接口地址。
* 后端 Controller 不直接写复杂 SQL。
* 数据库不关心页面如何展示。
* 通用工具函数不承载具体业务规则。

---

### 2.2 模块内聚

业务相关代码优先放在同一个业务模块下，而不是按技术类型散落到全局目录。

不推荐：

```txt
components/
  ProjectCard.tsx
api/
  projectApi.ts
hooks/
  useProjects.ts
types/
  project.types.ts
```

推荐：

```txt
features/
  project/
    components/
    api/
    hooks/
    types/
```

这样修改某个业务时，大部分代码都集中在同一个模块中。

---

### 2.3 边界明确

每个模块都应该明确：

```txt
我负责什么？
我不负责什么？
我依赖谁？
谁可以依赖我？
我暴露什么能力？
我隐藏什么实现细节？
```

边界不清晰会导致项目后期难以维护。

常见错误包括：

```txt
页面里写大量业务逻辑
组件里直接请求接口
全局 store 存放所有状态
utils 里塞业务规则
后端 controller 里写复杂业务逻辑
service 层直接返回数据库原始结构
接口返回结构不稳定
错误码没有统一约定
```

## 3. 前后端职责划分

### 3.1 前端职责

前端主要负责用户交互、页面展示、客户端状态和接口调用。

前端负责：

```txt
页面路由
页面布局
组件渲染
用户交互
表单校验
请求后端 API
展示 loading / empty / error 状态
管理客户端状态
处理用户权限下的页面可见性
适配响应式布局
前端埋点
前端异常捕获
```

前端不应该负责：

```txt
核心权限判断
核心业务数据校验
直接操作数据库
保存敏感密钥
绕过后端直接访问第三方私密服务
决定数据最终可信状态
处理复杂事务一致性
```

说明：

前端可以做表单校验，但这只是为了用户体验；后端仍然必须做完整校验。

---

### 3.2 后端职责

后端主要负责业务规则、数据校验、权限控制、数据持久化和系统集成。

后端负责：

```txt
API 接口设计
身份认证
权限校验
业务规则处理
数据校验
数据库读写
事务处理
文件处理
日志记录
错误处理
第三方服务集成
任务队列
定时任务
安全防护
接口限流
```

后端不应该负责：

```txt
页面样式
前端交互细节
按钮显示逻辑
前端路由跳转
纯 UI 状态
```

---

### 3.3 数据库职责

数据库负责数据持久化、基础约束和查询效率。

数据库负责：

```txt
表结构
字段类型
主键
外键
唯一约束
索引
事务
数据一致性
基础数据约束
```

数据库不应该负责：

```txt
复杂页面展示逻辑
前端状态管理
过多业务流程判断
用户交互逻辑
```

---

## 4. 推荐项目结构

推荐使用 monorepo 或前后端分离结构。

### 4.1 简单前后端分离结构

```txt
project-root/
├─ frontend/
├─ backend/
├─ docs/
├─ docker-compose.yml
├─ .env.example
└─ README.md
```

适合：

```txt
前端和后端技术栈不同
前后端部署相对独立
项目规模中等
团队分工明确
```

---

### 4.2 Monorepo 结构

```txt
project-root/
├─ apps/
│  ├─ web/
│  └─ api/
├─ packages/
│  ├─ shared/
│  ├─ config/
│  └─ types/
├─ docs/
├─ docker-compose.yml
├─ package.json
└─ README.md
```

适合：

```txt
前后端都使用 TypeScript
需要共享类型
需要共享工具函数
需要统一 lint / format / build
项目后期会拆多个应用
```

## 5. 前端架构规范

---

### 5.2 `src/app` 职责

`src/app` 是路由层，只负责页面入口、布局、路由级状态和 Next.js 约定文件。

`app` 目录可以包含：

```txt
page.tsx
layout.tsx
loading.tsx
error.tsx
not-found.tsx
route.ts
template.tsx
```

职责：

```txt
定义路由
定义页面入口
定义布局
定义 loading 状态
定义错误边界
定义 404 页面
定义轻量 Route Handler
```

不建议在 `app/page.tsx` 中写大量业务逻辑。

推荐写法：

```tsx
import { ProjectListView } from "@/features/project/components/ProjectListView";

export default function ProjectsPage() {
  return <ProjectListView />;
}
```

不推荐写法：

```tsx
export default function ProjectsPage() {
  // 大量请求逻辑
  // 大量业务判断
  // 大量表单逻辑
  // 大量状态处理
  // 大量 JSX
}
```

核心原则：

```txt
app 目录负责“页面在哪里”
features 目录负责“页面做什么”
```

---

### 5.3 `src/features` 职责

`features` 是前端业务模块层。

每个业务模块按业务领域拆分，例如：

```txt
features/
├─ project/
├─ document/
├─ auth/
├─ user/
└─ renderer/
```

每个模块内部推荐结构：

```txt
features/project/
├─ components/
├─ api/
├─ hooks/
├─ types/
├─ schemas/
├─ utils/
└─ index.ts
```

职责说明：

```txt
components/  当前业务模块专属组件
api/         当前业务模块接口封装
hooks/       当前业务模块 hooks
types/       当前业务模块类型定义
schemas/     当前业务模块校验规则
utils/       当前业务模块内部工具函数
index.ts     当前模块统一导出口
```

判断一个文件是否应该放进 `features/project`：

```txt
它是否只服务于 project 业务？
它离开 project 模块是否还成立？
修改 project 功能时是否大概率会修改它？
```

如果答案是“是”，就应该放在业务模块内部。

---

### 5.4 `src/components` 职责

`components` 只放全局通用组件，不放具体业务组件。

适合放在这里：

```txt
Button
Input
Select
Dialog
Table
Tabs
EmptyState
ErrorState
LoadingState
AppHeader
AppSidebar
PageContainer
```

不适合放在这里：

```txt
ProjectCard
ProjectForm
DocumentEditor
UserProfilePanel
RenderTaskList
```

判断标准：

```txt
这个组件脱离当前业务还能不能复用？
```

能复用，放 `components`。
不能复用，放 `features/xxx/components`。

---

### 5.5 `src/lib` 职责

`lib` 放底层能力封装和第三方库初始化。

适合放：

```txt
request.ts
env.ts
auth.ts
logger.ts
storage.ts
date.ts
upload.ts
```

示例：

```ts
// src/lib/request.ts

export async function request<T>(
  url: string,
  options?: RequestInit
): Promise<T> {
  const response = await fetch(url, {
    ...options,
    headers: {
      "Content-Type": "application/json",
      ...options?.headers,
    },
  });

  if (!response.ok) {
    throw new Error("Request failed");
  }

  return response.json();
}
```

业务模块基于 `lib/request.ts` 再封装具体接口：

```ts
// src/features/project/api/projectApi.ts

import { request } from "@/lib/request";
import type { Project } from "../types/project.types";

export function getProjects() {
  return request<Project[]>("/api/projects");
}
```

---

### 5.6 `src/stores` 职责

`stores` 放全局状态。

适合放全局状态：

```txt
当前登录用户
主题设置
全局弹窗
全局通知
侧边栏展开状态
跨页面共享状态
```

不适合放全局状态：

```txt
某个表单输入值
某个按钮 loading
某个弹窗内部开关
某个页面自己的筛选条件
某个组件 hover 状态
```

原则：

```txt
能放局部，就不要放全局。
能通过 URL 表达，就不要塞进 store。
能通过服务端数据重新获取，就不要重复存一份复杂缓存。
```

---

### 5.7 前端数据流

推荐数据流：

```txt
用户操作
  ↓
页面事件
  ↓
业务组件
  ↓
业务 hook
  ↓
业务 API
  ↓
底层 request
  ↓
后端接口
  ↓
返回数据
  ↓
更新状态
  ↓
重新渲染
```

示例：

```txt
src/app/projects/page.tsx
  ↓
src/features/project/components/ProjectListView.tsx
  ↓
src/features/project/hooks/useProjects.ts
  ↓
src/features/project/api/projectApi.ts
  ↓
src/lib/request.ts
```

禁止混乱数据流：

```txt
组件 A 改全局 store
组件 B 监听 store 后自动请求接口
组件 C 又修改 URL 参数
组件 D 又覆盖 store
最后没人知道数据从哪里来
```

## 6. 后端架构规范

后端推荐采用模块化单体架构。中小型项目不建议一开始就上微服务。

### 6.1 推荐后端目录结构

```txt
backend/
├─ src/
│  ├─ main.ts
│  ├─ app.ts
│  │
│  ├─ modules/
│  │  ├─ project/
│  │  │  ├─ project.controller.ts
│  │  │  ├─ project.service.ts
│  │  │  ├─ project.repository.ts
│  │  │  ├─ project.model.ts
│  │  │  ├─ project.dto.ts
│  │  │  ├─ project.schema.ts
│  │  │  └─ project.routes.ts
│  │  │
│  │  ├─ user/
│  │  ├─ auth/
│  │  └─ document/
│  │
│  ├─ common/
│  │  ├─ config/
│  │  ├─ database/
│  │  ├─ errors/
│  │  ├─ logger/
│  │  ├─ middleware/
│  │  ├─ response/
│  │  ├─ security/
│  │  └─ utils/
│  │
│  ├─ jobs/
│  ├─ scripts/
│  └─ tests/
│
├─ prisma/
│  ├─ schema.prisma
│  └─ migrations/
│
├─ .env
├─ .env.example
├─ Dockerfile
└─ package.json
```

如果不用 Prisma，可以替换为：

```txt
database/
├─ migrations/
├─ seeds/
└─ schema.sql
```

---

### 6.2 后端分层职责

推荐后端模块分层：

```txt
routes
  ↓
controller
  ↓
service
  ↓
repository
  ↓
database
```

#### routes

负责注册路由。

```txt
GET /projects
POST /projects
GET /projects/:id
PATCH /projects/:id
DELETE /projects/:id
```

不写业务逻辑。

---

#### controller

负责处理 HTTP 层逻辑。

职责：

```txt
读取请求参数
读取 query
读取 body
读取当前用户
调用 service
返回统一响应
```

不应该写复杂业务规则，不应该直接操作数据库。

---

#### service

负责业务逻辑。

职责：

```txt
业务规则
权限判断
数据校验
流程编排
事务控制
调用 repository
调用第三方服务
```

这是后端最核心的一层。

---

#### repository

负责数据访问。

职责：

```txt
数据库查询
数据库写入
封装 SQL / ORM 调用
屏蔽数据库细节
```

不应该处理 HTTP 请求，也不应该关心页面展示。

---

#### model / entity

负责描述数据模型。

```txt
Project
User
Document
RenderTask
```

---

#### dto / schema

负责接口数据结构和校验。

```txt
CreateProjectDto
UpdateProjectDto
ProjectResponseDto
```

---

## 7. API 接口规范

API 是前后端协作契约。接口不只是 URL，而是完整约定。

OpenAPI 规范提供了描述 RESTful API 的标准方式，使人和程序都能理解服务能力；项目较复杂时，建议使用 OpenAPI/Swagger 维护接口文档。

---

### 7.1 URL 命名规范

推荐使用资源名作为接口路径。

```txt
GET    /api/projects
POST   /api/projects
GET    /api/projects/:projectId
PATCH  /api/projects/:projectId
DELETE /api/projects/:projectId
```

不推荐：

```txt
GET /api/getProjectList
POST /api/createProject
POST /api/deleteProject
```

---

### 7.2 HTTP 方法约定

```txt
GET      查询数据
POST     创建数据
PATCH    局部更新数据
PUT      整体替换数据
DELETE   删除数据
```

---

### 7.3 请求参数约定

路径参数：

```txt
/api/projects/:projectId
```

查询参数：

```txt
/api/projects?page=1&pageSize=20&keyword=test
```

请求体：

```json
{
  "name": "Project A",
  "description": "Demo project"
}
```

---

### 7.4 响应结构约定

推荐统一响应结构：

```json
{
  "success": true,
  "data": {},
  "message": "ok",
  "requestId": "req_xxx"
}
```

失败响应：

```json
{
  "success": false,
  "error": {
    "code": "PROJECT_NOT_FOUND",
    "message": "Project not found",
    "details": {}
  },
  "requestId": "req_xxx"
}
```

说明：

```txt
success    表示请求业务是否成功
data       成功数据
message    成功提示，可选
error      错误对象
code       业务错误码
requestId  请求追踪 ID
```

---

### 7.5 分页约定

请求：

```txt
GET /api/projects?page=1&pageSize=20
```

响应：

```json
{
  "success": true,
  "data": {
    "items": [],
    "pagination": {
      "page": 1,
      "pageSize": 20,
      "total": 100,
      "totalPages": 5
    }
  }
}
```

---

### 7.6 排序和筛选约定

```txt
GET /api/projects?keyword=test&status=active&sortBy=createdAt&sortOrder=desc
```

约定：

```txt
keyword    搜索关键字
status     状态筛选
sortBy     排序字段
sortOrder  asc / desc
```

后端必须校验 `sortBy` 是否属于允许字段，不能直接拼接用户输入进 SQL。

---

### 7.7 错误码约定

错误码使用大写下划线风格。

示例：

```txt
VALIDATION_ERROR
UNAUTHORIZED
FORBIDDEN
PROJECT_NOT_FOUND
PROJECT_ALREADY_EXISTS
DATABASE_ERROR
INTERNAL_SERVER_ERROR
```

HTTP 状态码建议：

```txt
400 Bad Request          请求参数错误
401 Unauthorized         未登录
403 Forbidden            无权限
404 Not Found            资源不存在
409 Conflict             资源冲突
422 Unprocessable Entity 业务校验失败
500 Internal Server Error 服务端未知错误
```

## 8. Next.js Route Handler 使用约定

如果使用 Next.js 自带的 `app/api/**/route.ts`，应明确它的定位。

Next.js Route Handlers 用于在 `app` 目录中创建自定义请求处理逻辑，并且只在 `app` 目录下可用；它相当于 App Router 下的 API Routes。

适合放在 Next.js Route Handler 的逻辑：

```txt
BFF 层
轻量代理
登录回调
文件上传入口
Webhook
健康检查
简单聚合接口
```

不建议把大型后端业务全部塞进 Next.js Route Handler。

如果项目后端复杂，推荐独立后端服务：

```txt
frontend: Next.js
backend: NestJS / Express / Fastify / Spring Boot / Go / Django
database: PostgreSQL / MySQL
```

## 9. 鉴权与权限规范

### 9.1 认证 Authentication

```txt
你是谁？
```

常见方式：

```txt

```

### 9.2 授权 Authorization

授权解决的是：

```txt
你能做什么？
```

权限判断必须以后端为准。

前端可以控制按钮是否显示，但不能作为安全边界。

示例：

### 9.3 安全基础约定

基础安全要求：

```txt

敏感信息不能放前端
密码必须哈希存储
接口必须做权限校验
用户输入必须校验
文件上传必须限制类型和大小
错误信息不能泄露系统内部细节
接口需要限流
重要操作需要审计日志
```

## 14. Docker 与本地基础设施规范

### 14.1 Docker 的定位

Docker Compose 在本项目中主要用于本地开发环境和测试环境的基础设施编排。

它不只用于启动前端和后端，也用于启动数据库、中间件和依赖服务。

适合通过 Docker Compose 启动的服务包括：

```txt id="sidfy3"
关系型数据库：PostgreSQL / MySQL / MariaDB
图数据库：Neo4j
缓存：Redis
消息队列：RabbitMQ / Kafka
对象存储：MinIO
搜索引擎：Elasticsearch / OpenSearch
向量数据库：Qdrant / Milvus / Weaviate
监控组件：Prometheus / Grafana
日志组件：Loki / OpenSearch
```

原则：

```txt id="wzkxu8"
能用 Docker 稳定启动的基础设施，优先由 Docker Compose 管理。
业务代码不要和具体中间件启动方式强绑定。
不同项目可以按需启用不同中间件，但文档和结构保持通用。
```

---

### 14.2 推荐 Docker Compose 分层

推荐将 Docker Compose 分成三类服务：

```txt id="zb8qh6"
应用服务：frontend / backend / worker
基础设施：database / cache / graph-db / object-storage / message-queue
运维辅助：adminer / pgadmin / redis-commander / grafana
```

示例结构：

```txt id="lplkmm"
project-root/
├─ frontend/
├─ backend/
├─ docker/
│  ├─ postgres/
│  ├─ mysql/
│  ├─ neo4j/
│  ├─ redis/
│  ├─ minio/
│  └─ init/
├─ docker-compose.yml
├─ docker-compose.override.yml
├─ .env.example
└─ README.md
```

说明：

```txt id="oo1uin"
docker-compose.yml           通用服务定义
docker-compose.override.yml  本地开发覆盖配置
docker/                      中间件初始化脚本、配置文件、挂载目录
.env.example                 环境变量模板
```

### 14.3 通用 Docker Compose 示例

以下示例不是要求所有项目都必须启用全部服务，而是提供一个通用基础设施模板。

项目可以通过 Docker Compose profiles 按需启用。

```yaml id="gdyu1e"

  frontend:
    build:
      context: ./frontend
    ports:
      - "${FRONTEND_PORT:-3000}:3000"
    env_file:
      - ./frontend/.env.local
    depends_on:
      - backend
    profiles:
      - app

  backend:
    build:
      context: ./backend
    ports:
      - "${BACKEND_PORT:-3001}:3001"
    env_file:
      - ./backend/.env
    depends_on:
      - postgres
      - redis
    profiles:
      - app

  worker:
    build:
      context: ./backend
    command: npm run worker
    env_file:
      - ./backend/.env
    depends_on:
      - postgres
      - redis
    profiles:
      - worker

  postgres:
    image: postgres:16
    ports:
      - "${POSTGRES_PORT:-5432}:5432"
    environment:
      POSTGRES_USER: ${POSTGRES_USER:-app}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-app_password}
      POSTGRES_DB: ${POSTGRES_DB:-app_db}
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./docker/postgres/init:/docker-entrypoint-initdb.d
    profiles:
      - postgres
      - database

  mysql:
    image: mysql:8
    ports:
      - "${MYSQL_PORT:-3306}:3306"
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD:-root_password}
      MYSQL_DATABASE: ${MYSQL_DATABASE:-app_db}
      MYSQL_USER: ${MYSQL_USER:-app}
      MYSQL_PASSWORD: ${MYSQL_PASSWORD:-app_password}
    volumes:
      - mysql_data:/var/lib/mysql
      - ./docker/mysql/init:/docker-entrypoint-initdb.d
    profiles:
      - mysql
      - database

  neo4j:
    image: neo4j:5
    ports:
      - "${NEO4J_HTTP_PORT:-7474}:7474"
      - "${NEO4J_BOLT_PORT:-7687}:7687"
    environment:
      NEO4J_AUTH: ${NEO4J_USER:-neo4j}/${NEO4J_PASSWORD:-neo4j_password}
    volumes:
      - neo4j_data:/data
      - neo4j_logs:/logs
      - neo4j_import:/var/lib/neo4j/import
      - neo4j_plugins:/plugins
    profiles:
      - neo4j
      - graph

  redis:
    image: redis:7
    ports:
      - "${REDIS_PORT:-6379}:6379"
    volumes:
      - redis_data:/data
    profiles:
      - redis
      - cache

  minio:
    image: minio/minio
    command: server /data --console-address ":9001"
    ports:
      - "${MINIO_API_PORT:-9000}:9000"
      - "${MINIO_CONSOLE_PORT:-9001}:9001"
    environment:
      MINIO_ROOT_USER: ${MINIO_ROOT_USER:-minio}
      MINIO_ROOT_PASSWORD: ${MINIO_ROOT_PASSWORD:-minio_password}
    volumes:
      - minio_data:/data
    profiles:
      - minio
      - object-storage

  rabbitmq:
    image: rabbitmq:3-management
    ports:
      - "${RABBITMQ_PORT:-5672}:5672"
      - "${RABBITMQ_MANAGEMENT_PORT:-15672}:15672"
    environment:
      RABBITMQ_DEFAULT_USER: ${RABBITMQ_USER:-app}
      RABBITMQ_DEFAULT_PASS: ${RABBITMQ_PASSWORD:-app_password}
    volumes:
      - rabbitmq_data:/var/lib/rabbitmq
    profiles:
      - rabbitmq
      - message-queue

volumes:
  postgres_data:
  mysql_data:
  neo4j_data:
  neo4j_logs:
  neo4j_import:
  neo4j_plugins:
  redis_data:
  minio_data:
  rabbitmq_data:
```

### 14.9 数据卷规范

所有有状态中间件必须使用 volume 持久化数据。

需要持久化的服务包括：

```txt id="44gfus"
PostgreSQL
MySQL
Neo4j
Redis
MinIO
RabbitMQ
Elasticsearch
```

不应该把重要数据只放在容器内部。

推荐：

```yaml id="f0zqm9"
volumes:
  postgres_data:
  neo4j_data:
  redis_data:
  minio_data:
```

清理数据前必须明确影响。

```bash id="uxzh88"
docker compose down
```

只停止并移除容器，不会默认删除 volume。

```bash id="4k0ayj"
docker compose down -v
```

会删除 volume，数据库和中间件数据会被清空。

### 14.11 健康检查规范

核心服务建议配置健康检查。

示例：

```yaml id="yds3ca"
services:
  postgres:
    image: postgres:16
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d app_db"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
```

后端启动时也应该处理依赖未就绪的情况。

不要假设：

```txt id="yw601g"
容器启动 = 服务可用
```

正确理解：

```txt id="w73a01"
容器启动只代表进程开始运行。
服务可用需要通过健康检查或重试机制确认。
```

---

### 14.12 架构约束

禁止：

```txt id="x13vgd"
代码中硬编码中间件地址
代码中硬编码数据库账号密码
本地环境要求每个人手动安装所有中间件
不同开发者使用完全不同的数据库版本
将生产密钥写入 docker-compose.yml
将有状态服务数据只保存在容器内部
把所有中间件都强制启动，导致本地环境过重
```

推荐：

```txt id="oxqpd6"
用 Docker Compose 管理本地基础设施
用 profiles 按需启用中间件
用 .env 管理连接配置
用 .env.example 说明必需变量
用 volume 持久化中间件数据
用 migration 管理数据库结构
用健康检查和重试机制提升启动稳定性
```

```txt
8. 补充测试
9. 代码审查
10. 部署上线
11. 观察日志和错误
```

不要直接从写页面开始。

推荐顺序：

```txt
模型先行
接口先行
边界先行
实现随后
```

## 19. 架构红线

以下行为应尽量避免：

```txt
页面组件超过 500 行仍不拆分
一个 utils 文件放几十个无关函数
全局 store 管理所有页面状态
接口没有统一响应格式
接口错误直接返回数据库错误
前端直接依赖数据库字段结构
后端直接返回 ORM 原始对象
controller 写复杂业务逻辑
service 层混入 HTTP request / response
repository 层处理权限逻辑
环境变量没有 .env.example
没有 requestId 无法追踪问题
没有 README 无法启动项目
```

---

## 20. 最小可执行架构标准

一个合格的中小型项目，至少应该具备：

```txt
清晰的 frontend / backend 目录
清晰的前端 app / features / components / lib 分层
清晰的后端 modules / common 分层
统一 API 响应结构
统一错误码
统一环境变量示例
统一数据库迁移方式
统一日志格式
基础鉴权和权限校验
Docker Compose 本地启动
README 启动说明
```

## 21. 总结

本架构规范的核心思想是：

```txt
前端负责展示和交互
后端负责业务和数据
数据库负责持久化和约束
API 负责前后端契约
模块负责业务内聚
分层负责复杂度控制
约定负责团队协作
```

判断架构是否健康，可以看这几个问题：

```txt
新增功能时，应该改哪里是否清楚？
出现问题时，应该查哪里是否清楚？
一个模块的代码是否集中？
前端是否依赖了不该依赖的后端细节？
后端是否混入了页面展示逻辑？
接口结构是否稳定？
错误是否可追踪？
项目是否能被新人快速启动？
```

如果这些问题都有明确答案，说明项目架构已经比较健康。
